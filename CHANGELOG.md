## [Unreleased]

### Fixed
- **`error_message` obsolète après une reprise réussie** : la colonne `jobs.error_message` n'était jamais effacée — un vieux message (ex. « VRAM insuffisante pour cohere-summary ») restait collé après qu'un job avait repris et progressé (state `ready_to_process`/`completed`), trompant l'UI. Nouvel invariant dans `JobStore.update_state` : **`error_message` non vide ⟺ `state == FAILED`** — toute transition hors FAILED l'efface. Le détail par exécution reste tracé dans `extra_data.execution.last_error` (inchangé).
- **Job bloqué indéfiniment derrière notre propre LLM d'arbitrage chaude (admission)** : la récupération VRAM (`_reclaim_vram_from_idle_arbitrage_llm`) ne se déclenchait qu'**en cours de phase** (post-dispatch). Or l'**admission** du scheduler (`_resources_available` → `can_allocate`) refusait de dispatcher un job tant que la VRAM libre était insuffisante — sans savoir que l'occupant était **notre propre LLM d'arbitrage inactive** (gardée chaude, étalée sur les GPU). Le job restait en `waiting` pour toujours, le reclaim ne tournant jamais. Désormais, à l'admission, quand une phase **non-LLM** (STT/diarisation — la phase `llm_arbitration` est déjà ignorée via `llm_shared`) manque de VRAM, le scheduler **arrête proprement notre LLM d'arbitrage inactive** (catégorie 1, verrou LLM libre, toujours autorisée — relancée à la phase de correction) puis re-teste l'allocation. Logique mutualisée avec le runner via `transcria/gpu/vram_reclaim.stop_idle_arbitrage_llm`.

### Added
- **Politique de préemption VRAM `gpu.preemption`** (`own-only` par défaut | `aggressive`) — réglable dans l'**éditeur de configuration** (`/admin/config`, section « Ressources GPU »). `own-only` (recommandé, infra partagée) : n'arrête que **nos** process gérés inactifs (LLM d'arbitrage trackée), **jamais** un process tiers. `aggressive` : préempte aussi les serveurs d'inférence tiers (`kill_patterns`, process non trackés) — **uniquement dans la fenêtre calendaire `force_gpu`** —, à réserver à un GPU dédié à TranscrIA. Remplace l'appel `force_free_gpu` inconditionnel-en-fenêtre de l'admission par cette politique explicite (défaut plus conservateur qu'avant : plus de kill de process tiers hors `aggressive`). `_run_quick_transcription` réservait toujours la VRAM locale (`summary_stt`, 6000 Mo via `_gpu_session`) **même quand le STT du résumé est routé vers un nœud de ressources distant** (`inference.mode` ∈ remote/hybrid). Sur un tier sans GPU, cela provoquait une fausse contention / une mise en attente VRAM à tort. Désormais le résumé saute la réservation locale quand `summary_stt` est une phase distante (`WorkflowRunner._phase_runs_remotely`), alignant le résumé sur le pipeline principal (`_reserve_gpu_phase`). Cf. `docs/SERVICE_RESSOURCES_GPU.md` §9.
- **Échec silencieux de la LLM d'arbitrage (résumé corrompu sans alerte)** : si opencode se terminait « avec succès » mais **sans produire de texte** (« 0 textes » — cause fréquente : transcript trop long), il ne réécrivait pas `summary.md` et le **placeholder** de `SummaryGenerator` était relu, parsé, puis stocké comme s'il était le résumé (`meeting_context["summary_llm"]` corrompu), le job passant `SUMMARY_DONE` en silence (incident `e62295c1`). Désormais : `OpenCodeRunner.run_summary` détecte la non-production de façon robuste (mtime de `summary.md` **avant/après** le run, plus de matching de chaîne fragile) et expose `_summary_produced`. `WorkflowRunner._run_llm_summary` **retente la seule phase LLM jusqu'à 3 fois** (LLM déjà chargée : pas de re-STT). Après 3 échecs : `meeting_context` **non corrompu**, job **non** `SUMMARY_DONE` (drapeau `extra_data.summary_llm_failed`), bandeau d'erreur dans le wizard et **résumé relançable** — la relance **réutilise le transcript en cache** (`quick_transcript.txt`/`summary.json`) au lieu de refaire le STT GPU.
- **VRAM bloquée par notre propre LLM d'arbitrage inactive** : après un résumé synchrone, la LLM d'arbitrage restait chargée (aucun idle-stop) et, étalée sur les GPU via `--tensor-split`, empêchait un STT de réserver sa VRAM — bloquant indéfiniment la reprise `waiting_vram`. Désormais, sur pénurie VRAM d'un STT (résumé/transcription/diarisation), `WorkflowRunner._reclaim_vram_from_idle_arbitrage_llm` **arrête notre LLM d'arbitrage uniquement si elle est inactive** (verrou LLM libre — garde-fou multi-job : jamais une LLM en service) puis **retente une fois** la réservation. C'est notre process géré (jamais un process tiers). La LLM reste sinon chaude pour le job suivant.

### Added
- **`transcria doctor --llm-smoke`** : test **opt-in** qui lance réellement opencode → LLM avec une consigne triviale et vérifie qu'il **produit du texte** — attrape a priori la panne « 0 texte » ci-dessus (`check_opencode_smoke`, réutilise la capture de log d'arbitrage). Nécessite la LLM up et consomme de la VRAM : volontairement hors du préflight par défaut (qui reste GPU-free et sans effet de bord).

### Changed
- **VRAM insuffisante → mise en attente au lieu d'échec terminal** : quand la VRAM locale manque pour une phase GPU (STT rapide cohere-summary, transcription, diarisation, détection de locuteurs), le job ne passe plus en **FAILED** (état terminal + e-mail d'échec au propriétaire + alerte rouge). Une indisponibilité VRAM est *transitoire* : le job est désormais marqué **« en attente de VRAM »** (statut d'exécution `waiting_vram`, non terminal) et **reprend automatiquement** dès que la mémoire se libère — sur le même principe que le mode `deferred` des ressources distantes (§7.2). Côté **file principale** : `WorkflowRunner` (transcription/diarisation/locuteurs) et `PipelineService` remontent un signal `vram_wait` ; `JobExecutorService._run_process` re-queue le job (`QueueStore.requeue_later`) sans état terminal et le scheduler le redispatche dès admission VRAM possible. Côté **résumé synchrone** (« Générer le résumé ») : la 1ʳᵉ tentative reste synchrone ; sur `vram_wait`, `api_summary` restaure l'état pré-résumé et **enfile une reprise serveur** (mode de file `summary`, profil VRAM dédié) que le **scheduler reprend automatiquement dès libération de la VRAM, même sans page ouverte**. `JobExecutorService._run_process` exécute alors `run_summary` (qui gère l'état `SUMMARY_DONE`/`FAILED`) sans marquer `COMPLETED` ni notifier le propriétaire. Le wizard affiche un bandeau d'attente et **poll `GET /status`** jusqu'à `summary_done` (plus de re-POST client → aucune double-exécution). Dans tous les cas, **l'administrateur est alerté une seule fois** par épisode (e-mail `send_admin_vram_alert_async` aux comptes ADMIN actifs + log `WARNING` structuré + bandeau in-app comptant les jobs `waiting_vram`). TranscrIA continue de **ne jamais tuer** un process GPU tiers (le bridage de `force_free_gpu` est inchangé). Voir `docs/SERVICE_RESSOURCES_GPU.md` §7.2.

### Fixed
- **Invitation de réunion : affichage fiabilisé (pas une perte de données)** : le brief d'invitation collé était bien sauvegardé (`extra_data.meeting_invite.brief`), mais le `<textarea>` était toujours rendu vide et le badge de confirmation ne s'affichait que si des **noms** étaient extraits d'e-mails `prenom.nom@` — sans e-mail parsable, un brief enregistré semblait perdu. Désormais : le textarea est **pré-rempli** avec le brief stocké, le badge « Invitation enregistrée » s'affiche dès qu'un **brief OU des noms** existent, et le collé est **auto-sauvegardé au `blur`** (plus seulement au clic « Générer le résumé ») pour survivre à une étape échouée/abandonnée.
- **Diagnostic d'une LLM d'arbitrage qui ne démarre pas** : le lancement du serveur LLM (`VRAMManager.launch_arbitrage_llm`, `ScriptLLMBackend.ensure_available`) redirigeait stdout/stderr vers `/dev/null`, puis se contentait d'attendre l'ouverture du port jusqu'à **600 s** avant de logguer un timeout sec. Conséquence (constatée sur une autre machine après un `git pull`) : quand le serveur échouait au démarrage (chemin de binaire faux, OOM GPU, `--tensor-split` incompatible avec le nombre de GPUs…), la LLM restait « down » **sans aucun log expliquant pourquoi** — seul restait le placeholder « Résumé indisponible (LLM non configurée) ». Désormais : la sortie du script est **capturée** dans `services.arbitrage_log_path` (défaut `/tmp/arbitrage_llm_<port>.log`) ; l'attente du port **détecte la mort précoce du process** (`proc.poll()`) et abandonne immédiatement au lieu d'attendre tout le timeout ; en cas d'échec (mort précoce **ou** timeout), le code de sortie et les **dernières lignes du log** sont loggués en `ERROR`. Aligne le backend LLM sur la convention déjà utilisée par le superviseur STT. Voir `docs/CONFIG_REFERENCE.md` (note d'exploitation « Diagnostic d'un démarrage LLM raté »).
- **Enqueue idempotent face à une course double-submit** : deux requêtes simultanées sur le **même** `job_id` (double-clic / session partagée) pouvaient toutes deux passer les gardes (`is_execution_active`/`already_active`, lecture avant écriture) puis tenter l'`INSERT` ; la contrainte unique `job_queue.job_id` préservait déjà la correction (pas de doublon ni de double-run) mais le perdant levait une `IntegrityError` non capturée → 500. `QueueStore.enqueue` capture désormais l'`IntegrityError`, fait un `rollback`, relit l'entrée gagnante et la réutilise (helper partagé `_refresh_entry`) — course gracieuse, plus de 500. Une `IntegrityError` sans rapport avec l'unicité `job_id` est re-levée (pas de masquage).
- **Fourchette de locuteurs honorée en diarisation distante** : la contrainte de locuteurs par job (`speaker_hint` → `num_speakers`/`min_speakers`/`max_speakers`) était **silencieusement ignorée** quand `models.diarization_backend=remote` (régression vs mode local). Elle est désormais transmise de bout en bout : `RemoteDiarizer` lit `_effective_speaker_params()` → `InferenceClient.diarize(..., speaker_params)` (JSON en file_ref, champs de formulaire en upload) → route `/infer/diarize` (extraction + validation) → `DiarizeEngine.diarize(speaker_params=…)` → `DiarizerService.diarize_audio(speaker_params=…)`, où la contrainte par appel **prime sur la config statique du nœud** (re-validée au bord réseau via le validateur partagé `_normalize_speaker_params`). Parité local↔distant rétablie.
