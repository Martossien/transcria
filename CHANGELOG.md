## [Unreleased]

### Added
- **Base de données PostgreSQL (Phase A)** : bascule de SQLite vers **PostgreSQL** (driver `psycopg`) pour tenir la concurrence. Schéma géré par **Alembic** (`alembic upgrade head` lancé par `start.sh` ; `db.create_all()` réduit au bootstrap dev/tests) ; test anti-dérive `tests/test_alembic_migrations.py`. DSN via `TRANSCRIA_DATABASE_URL` (prioritaire, garde le mot de passe hors config versionnée), pool dialecte-aware. Colonnes temporelles en `TIMESTAMPTZ`, `audit_logs.actor_id` en `ON DELETE SET NULL`. Migration des données : `scripts/migrate_sqlite_to_postgres.py`. Tests sur cluster PostgreSQL éphémère (`pytest-postgresql`). SQLite reste un repli mono-process pour le dev. Voir `docs/INSTALL.md` §7.
- **Montée en charge & concurrence (Phase B)** : exploite PostgreSQL pour scaler sans double-exécution ni sur-réservation VRAM. Conception complète dans `docs/CONCURRENCE_ET_CHARGE_PHASE_B.md`.
  - **Rôles de process** `runtime.role` / `TRANSCRIA_ROLE` ∈ `all` (tout-en-un, défaut) | `web` (tier HTTP sans état, **gunicorn -w N**, `wsgi.py`) | `scheduler` (orchestrateur unique). Garde-fou « ordonnanceur unique » par **verrou consultatif PostgreSQL** (`transcria/queue/scheduler_lock.py`). Unités systemd `deploy/transcria-{migrate,web,scheduler}.service` + `deploy/nginx-transcria.conf.example`.
  - **Claim de job atomique** `FOR UPDATE SKIP LOCKED` (`QueueStore.claim`) : pas de double-dispatch même multi-worker ; capacité lue en base (`count_running`).
  - **Failover actif/passif des nœuds de ressources** : `inference.nodes` (liste ordonnée), `FailoverInferenceClient` qui bascule par priorité sur `InferenceUnavailable` (pas sur 4xx), recalculé par appel (pas de split-brain).
  - **Admission VRAM-aware + backpressure à l'échelle** : aging ensembliste (un seul `UPDATE`), capacité du nœud au dispatch (`available_remote_slots`), admission selon le coût VRAM local/distant (`_resources_available` + `remote_vram_admits`), index partiel `ix_job_queue_waiting_order`, cache court de `GET /api/resources/status` (`inference.resilience.capabilities_cache_ttl_s`).
  - **Profil de concurrence & observabilité du goulot** (`transcria/workflow/concurrency_profile.py`) : classe les étapes *serial*/*delegated*, mesure le % sériel et l'étape goulot, expose une attente estimée dans `GET /api/resources/status` (clé `concurrency`). Surcharges via `workflow.concurrency_profile`.
  - **Réveil instantané optionnel** `LISTEN/NOTIFY` (`workflow.queue.use_listen_notify`, `transcria/queue/notify_listener.py`) : le polling reste le filet de sûreté.
  - Durcissement du nœud de ressources : `ensure_ready` sérialisé par moteur, état de charge dans `/capabilities` (`capacity/inflight/queued/last_wait_s`, `ensure_in_progress`), invariant single-process documenté.
- **Qualification du son avant transcription (axe 1)** : caractérisation acoustique enrichie pour mieux décider de la difficulté d'un audio. `squim_scorer.py` (STOI/PESQ/SI-SDR via torchaudio), `dnsmos_scorer.py` (SIG/BAK/OVRL P.835 via onnxruntime, modèle ONNX embarqué), `acoustic_metrics.py` (RT60 sur décroissances libres, C50, SNR fenêtré, codec VoIP), fusion pondérée `difficulty_map.py` (avec veto overlap). Branchés dans le préflight (globaux + par fenêtre, lazy). Encart diagnostic dans le wizard (scores SQUIM/DNSMOS, conseil bruit-vs-parole, résumé temporel). Actifs par défaut via `workflow.audio_preflight.{squim,dnsmos,acoustic}`. Attribution : `THIRD_PARTY_NOTICES.md` (poids CC-BY-4.0).
- **Rapport Word (.docx) adapté au type de réunion** : document professionnel généré en fin de workflow (téléchargeable / inclus au ZIP). Extraction structurée par la LLM (décisions, actions, points bloquants/reportés, votes, résolutions, ordre du jour) avec dégradation gracieuse ; **18 types de réunion** (CSE, CODIR/COMEX, point projet, réunion client, crise…) à champs et thèmes visuels dédiés (calcul du quorum CSE, badge confidentiel/crise). Voir `docs/FEATURE_DOCX_REPORT.md`.
- **Inférence distante & autonomie VRAM du STT (v1)** : TranscrIA peut opérer en frontale dont les ressources GPU (STT, diarisation, empreinte vocale) tournent sur un nœud distant — ou en tout-en-un. Conception complète dans `docs/SERVICE_RESSOURCES_GPU.md`.
  - STT distant via protocole **OpenAI** `/v1/audio/transcriptions`, moteur de serving **non hardcodé** (vLLM/SGLang/custom) : `RemoteTranscriber` + `AsrClient` ; sélection par `inference.mode` + `inference.stt.backends[*].url` ; `response_format` par backend ; conversion WAV 16k mono (contourne le bug MP3) ; concurrence par tour (`inference.stt.concurrency`).
  - Autonomie VRAM du STT (cycle A/B/C, non intrusif) : `SttVramPlanner` (pré-check fraction×total vLLM + relocalisation GPU), `SttEngineSupervisor`, route `POST /engines/ensure` (lance à la demande / 503 si saturé). Détection via `GET /capabilities`.
  - Admission en mode dégradé (§7.2) : `resource_gate.prepare_remote_resources()` en pré-vol de `PipelineService.run_process` (poursuit / file / échec explicite borné par `inference.resilience.max_unavailable_s`) — jamais d'échec silencieux ni de spin. Panneau d'état : `GET /api/resources/status` + carte dans `dashboard_status.html`.
  - Correctif : une phase servie à distance ne réserve plus de VRAM locale (`WorkflowRunner._phase_runs_remotely`).
  - **Re-queue différé (§7.2)** : un job dont les ressources distantes sont transitoirement injoignables est replanifié (`QueueStore.requeue_later` + `scheduled_at`) au lieu d'échouer ; il re-tente après backoff jusqu'au succès ou échec explicite au-delà de `inference.resilience.max_unavailable_s`. `job_executor` gère le verdict `deferred` (état `queued`, pas de FAILED).
  - **Idle-stop minimal (v1.2)** : un moteur STT externe déclaré avec `idle_timeout_s > 0` et inactif au-delà du seuil est arrêté (`SttEngineSupervisor.reap_idle` via `scripts/stop_stt.sh`), de façon opportuniste (poll `/capabilities` + `ensure_ready`, sans tâche de fond). Non intrusif : ne touche que les moteurs servis par le service. Défaut `idle_timeout_s=0` = résident.
  - Scripts : `_stt_serve_lib.sh`, `launch_stt_{cohere,whisper,granite}.sh`, `stop_stt.sh`, `test_stt.sh`, `scripts/smoke_remote_stt.py`. Banc E2E : options `--remote-stt` / `--remote-inference`.
- Architecture diarisation modulaire : `BaseDiarizer` (ABC, `transcria/stt/base_diarizer.py`) factorise la logique partagée entre tous les backends — cache, extraits audio, empreintes acoustiques, fingerprint, chargement GPU. `DiarizerService` (pyannote) et `SortformerDiarizer` (NeMo) héritent de cette classe de base. `create_diarizer(config, device)` et `get_diarizer_vram_mb(backend, config)` centralisent la construction et la réservation VRAM dans `transcria/stt/diarizer_factory.py`. `list_available_backends()` retourne la liste des backends enregistrés (`pyannote`, `sortformer`). `runner.py` et `speaker_detection.py` utilisent désormais la factory au lieu d'instancier `DiarizerService` directement.

Activé via `models.diarization_backend=sortformer`. `_parse_sortformer_output()` convertit les lignes `"start end speaker_N"` produites par `model.diarize()` en dict `{start, end, speaker, duration}`. `_normalize_speaker_id()` normalise `speaker_0` → `SPEAKER_00`.
