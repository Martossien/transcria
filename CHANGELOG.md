### Added
- **Qualification du binaire `llama-server` (version réelle / bibliothèques / CUDA) — fin des échecs silencieux au runtime.** L'install ne faisait que *trouver* un binaire (3 chemins en dur, premier exécutable gagné) sans vérifier qu'il **chargerait** nos modèles — trois trous : (1) **version** : les archis `gated-delta`/`gemma4` (profils 12→64 Go) exigent llama.cpp ≥ b9630, mais un binaire trop vieux échoue *silencieusement* (« unknown model architecture ») ; PIÈGE découvert : le numéro de `--version` est **non fiable** — un binaire **réellement b9632** compilé depuis un clone git se déclare `version: 579` (compteur de build faux). (2) **bibliothèques** : un build maison résout ses `.so` par RPATH (env conda) — env déplacé/supprimé ⇒ `ldd` « not found » ⇒ le serveur ne démarre pas, alors que le fichier est bien exécutable (mode d'échec le plus courant). (3) **recherche étroite** : `/opt`, `~/ik_llama.cpp`, envs conda non trouvés. Nouveau module **pur** `transcria/gpu/llama_runtime.py` (parseurs `--version`/`git describe`/`ldd`, `evaluate_runtime`) : résout la **vraie version via l'arbre git** (source autoritaire) et ne traite le self-report que comme signal *mou*. **`critical` est réservé à ce qui empêche le binaire de DÉMARRER** : seules les **libs manquantes** y donnent (indépendant du modèle — RPATH/conda cassé). **Toute version trop basse n'est qu'un avertissement** — confirmée par git *ou* self-report — car le besoin ≥ b9630 est **relatif au modèle chargé** (les archis gated-delta/gemma4 l'exigent ; un autre modèle tourne sur une version plus vieille) et de toute façon **rien n'est bloquant** (l'install signale mais propose toujours le binaire ; le check ne fait qu'afficher). Build CPU-only ⇒ avertissement. Un fork au schéma de versions distinct (ik_llama : tags `tNNNN`, pas de `bNNNN`) n'est jamais pénalisé pour l'absence de `bNNNN`. CLI `scripts/detect_llama_server.py` (recherche élargie, sorties human/shell/json, codes 0/2/1) : branché dans `install.sh` (repli défensif sur l'ancienne boucle) et ajouté en fin de `scripts/check_arbitrage_llm.sh`. Escape-hatch `LLAMA_LD_LIBRARY_PATH` honoré par les 6 profils (`scripts/arbitrage_profiles/`) pour un env de libs non standard. Logique **entièrement testée sans binaire ni GPU** (`tests/test_llama_runtime.py`, dont le piège b9632→579 et la primauté de `git describe`).
- **Planification du placement LLM par carte (fin du choix « à la somme ») + contrôle de calibration mesuré.** L'install choisissait le palier sur la VRAM **totale** (`recommend_llm_tier(GPU_VRAM_TOTAL_MB)`) puis appliquait un profil au placement **figé** (mono-GPU pour 12/16/24, split fixe `[0,1]`/`[0,1,2]` pour 32/48/64) — ce qui OOM sur les topologies courantes : *2× 8 Go* (palier 16 mono sur une carte de 8), *cartes hétérogènes* (8+24 → split égal 15,8 Go sur la carte de 8), *1× 5090* (palier 32 splité sur une seule carte), ou *2× 5090* (palier 64 écrit `[0,1,2]` pour 2 cartes). Nouveau module **pur** `transcria/gpu/llm_placement.py` : table d'**empreintes mesurées** par palier (poids + KV au contexte + compute, cf. `docs/BENCH_LLM_PALIERS.md`) et logique qui choisit **le plus grand palier réellement plaçable** sur la topologie réelle (vérifie chaque carte, mono ou split égal, marge plancher 1,5 Go calée sur le bench), **jamais un placement qui OOM** ; sinon « transcription brute » avec un **indice actionnable** (« vos 2 cartes hébergeraient le modèle 16 en split, profil --tensor-split personnalisé requis »). `install.sh` s'appuie dessus (repli défensif sur la table par somme si le planner échoue) et écrit la **vraie calibration par carte** (`llm_vram_mb` / `llm_gpu_indices` / `llm_vram_mb_per_gpu`) via `transcria/config/gpu_calibration.py` (round-trip **ruamel**, atomique, préserve commentaires/secrets — là où le `sed` de `switch_arbitrage_llm.sh` échouait silencieusement sur les listes YAML par blocs). CLI `scripts/plan_llm_placement.py` (`plan` / `verify`) : `verify` **mesure** la VRAM réellement consommée par carte (nvidia-smi, ancrage UUID) et la compare au déclaré → signale dérive (calibration périmée), marge critique (OOM imminent, ex. quant Q4_K_M @0,2 Go), carte déclarée inutilisée, débordement hors placement ; greffé en fin de `scripts/check_arbitrage_llm.sh` (au passage : bug `max_tokens: 5` corrigé — un modèle *reasoning* dépense ses premiers tokens dans `<think>`, `text` restait vide et concluait à tort à une panne). Trois renforts : (1) le fix ci-dessus ; (2) **log throttlé** côté scheduler (`remote_gpu_data_missing`, prédicat pur) — un `WARNING` émis **une fois au changement d'état** « nœud joignable mais sans donnée GPU » (pas à chaque tick) ; (3) **check d'installation** `doctor.check_inference_node_gpus` : un nœud distant joignable qui n'énumère aucun GPU via `/capabilities` → `WARN` clair (détecté à l'install, pas via des jobs qui stagnent en prod). - **Web/auth : durcissement sécurité (cookie SameSite/Secure, open redirect, logout POST) — revue du sous-système** — points sains confirmés (hachage werkzeug, autorisation `@requires(Permission)`, IDOR job via `_can_access_job`, pas de `shell=True`, pas de path-traversal `job_id`). Quatre durcissements (audit) : **(1) anti-CSRF sans jeton** : `SESSION_COOKIE_SAMESITE="Lax"` (+ `HTTPONLY=True` explicite) — le cookie de session n'est plus envoyé sur un POST cross-site, ce qui neutralise le CSRF sur les routes mutantes (création d'admin, suppression de job, sauvegarde de config) que les décorateurs `@requires` ne protégeaient pas (le CSRF emprunte la session autorisée de la victime). - **Stockage/SQL : race *lost-update* sur `extra_data` corrigée (verrou de ligne) + WAV intermédiaires exclus de la synchro pg (revue du sous-système)** — push/pull jugés sains (transaction par fichier, sha256 vérifié, matérialisation atomique, conflits non écrasés, mtime non préservé → cohérent avec le content-sha de la reprise). Deux correctifs : **(1)** `JobStore.update_extra_data` faisait un read-modify-write **sans `FOR UPDATE`** ; or `extra_data_json` est partagé entre tiers/threads (frontale : annulation/contexte ; scheduler : progression toutes les quelques secondes, marqueurs de reprise). Deux écritures concurrentes du même job se perdaient — scénario réel : une **demande d'annulation écrasée** par une mise à jour de progression (l'updater `{**extra}` réécrit tout sur une lecture périmée) → le job continue malgré l'annulation. Vaut en all-in-one (deux threads, sessions distinctes) et en pg multi-tiers (deux process). Correctif : `with_for_update()` + `populate_existing` sérialise et relit la valeur committée sous le verrou (motif éprouvé de `QueueStore.claim()`). - **Politique de préemption VRAM `gpu.preemption`** (`own-only` par défaut | `aggressive`) — réglable dans l'**éditeur de configuration** (`/admin/config`, section « Ressources GPU »). `own-only` (recommandé, infra partagée) : n'arrête que **nos** process gérés inactifs (LLM d'arbitrage trackée), **jamais** un process tiers. `aggressive` : préempte aussi les serveurs d'inférence tiers (`kill_patterns`, process non trackés) — **uniquement dans la fenêtre calendaire `force_gpu`** —, à réserver à un GPU dédié à TranscrIA. Sur un tier sans GPU, cela provoquait une fausse contention / une mise en attente VRAM à tort. Désormais le résumé saute la réservation locale quand `summary_stt` est une phase distante (`WorkflowRunner._phase_runs_remotely`), alignant le résumé sur le pipeline principal (`_reserve_gpu_phase`). ### Changed
- **VRAM insuffisante → mise en attente au lieu d'échec terminal** : quand la VRAM locale manque pour une phase GPU (STT rapide cohere-summary, transcription, diarisation, détection de locuteurs), le job ne passe plus en **FAILED** (état terminal + e-mail d'échec au propriétaire + alerte rouge). Une indisponibilité VRAM est *transitoire* : le job est désormais marqué **« en attente de VRAM »** (statut d'exécution `waiting_vram`, non terminal) et **reprend automatiquement** dès que la mémoire se libère — sur le même principe que le mode `deferred` des ressources distantes (§7.2). Côté **file principale** : `WorkflowRunner` (transcription/diarisation/locuteurs) et `PipelineService` remontent un signal `vram_wait` ; `JobExecutorService._run_process` re-queue le job (`QueueStore.requeue_later`) sans état terminal et le scheduler le redispatche dès admission VRAM possible. Côté **résumé synchrone** (« Générer le résumé ») : la 1ʳᵉ tentative reste synchrone ; sur `vram_wait`, `api_summary` restaure l'état pré-résumé et **enfile une reprise serveur** (mode de file `summary`, profil VRAM dédié) que le **scheduler reprend automatiquement dès libération de la VRAM, même sans page ouverte**. `JobExecutorService._run_process` exécute alors `run_summary` (qui gère l'état `SUMMARY_DONE`/`FAILED`) sans marquer `COMPLETED` ni notifier le propriétaire. Le wizard affiche un bandeau d'attente et **poll `GET /status`** jusqu'à `summary_done` (plus de re-POST client → aucune double-exécution). Dans tous les cas, **l'administrateur est alerté une seule fois** par épisode (e-mail `send_admin_vram_alert_async` aux comptes ADMIN actifs + log `WARNING` structuré + bandeau in-app comptant les jobs `waiting_vram`). TranscrIA continue de **ne jamais tuer** un process GPU tiers (le bridage de `force_free_gpu` est inchangé). Voir `docs/SERVICE_RESSOURCES_GPU.md` §7.2.
